# 电瓶导购站设计思路

**日期：** 2026-05-15  
**域名方向：** battery.sgcarpass.com（二级域名）  
**技术：** Nuxt 3（SSR）/ Vercel 免费托管 / 复用现有 RuoYi 后端

---

## 一、核心理念：从圆心出发

**唯一考核指标：到店核销数量。**

链路：Google/AI 搜到 → 车型页 → 领 WA 优惠券 → 到店核销 → 修理工佣金

搜索引擎或 AI 找不到，这个网站做再好也没用。  
所以第一要务是让 Google 和 AI（ChatGPT / Perplexity / Google AI Overview）能找到、能引用。

---

## 二、第一圆：让搜索引擎/AI 找到你

### 车主搜什么？

车主不搜 "55D23L battery"，他搜：
- "Toyota Vios battery replacement Singapore"
- "Honda Jazz battery price Singapore"
- "Proton Saga battery how long"

**结论：以车型为主轴建页面，电瓶型号是附属信息。**

### URL 结构

每个车型一个独立 URL：

```
battery.sgcarpass.com/car/toyota-vios
battery.sgcarpass.com/car/honda-jazz-2020
battery.sgcarpass.com/car/hyundai-avante
```

### AI 搜索（ChatGPT / Perplexity / Google AI Overview）

AI 的逻辑和 Google 相似——要的是清晰的结构化事实：

> "Toyota Vios (2014-2023) 需要 55D23L，12V 45Ah，新加坡热带气候推荐 EFB 或 AGM，平均寿命 2-3 年。"

这种句子，AI 会直接引用。**FAQ 结构**对 AI Overview 极重要，AI 喜欢抓问答格式。

## 三、第二圆：每个车型页的内容结构

```
H1: Toyota Vios Battery Replacement Singapore (2014–2023)
├── 适配电瓶规格表（型号 / 电压 / 容量 / 推荐类型）
├── 新加坡气候下为何推荐 AGM/EFB（内容，建立信任）
├── 价格区间（SGD）
├── FAQ
│     "多久换一次？"
│     "有没有上门服务？"
│     "EFB 和 AGM 区别？"
├── 领优惠券 CTA → 输手机号 → WhatsApp 自动发券
└── 附近门店地图
```

FAQ 是最重要的一节，Google Featured Snippet 和 AI Overview 都从这里抓内容。

## 四、第三圆：技术架构

```
Vercel (免费)
  └── Nuxt 3 (SSR，对 SEO 友好)
        └── 调用现有 RuoYi 后端 API
```

### 现有后端需新增的接口（轻量）

| 接口 | 说明 |
|---|---|
| `GET /api/pub/car-battery/{slug}` | 返回车型信息 + 适配电瓶规格 |
| `POST /api/pub/coupon/request` | 车主输手机号 → 生成并发 WA 优惠券 |
| `POST /api/biz/coupon/redeem` | 修理工核销 → 记录佣金 |

**优惠券派发直接走现有 WhatsApp 通道**——复用已有基础设施，改动最小。

## 五、第四圆：内容护城河

新加坡市面主流车型约 150-200 款，每款一页。  
这是最重的工作，也是最难被竞争对手复制的资产。

### 分阶段计划

| 阶段 | 目标 | 说明 |
|---|---|---|
| 第一批 | Top 30 车型 | Vios、Jazz、Civic、Sylphy、Attrage 等销量大、搜索量高的 |
| 持续补充 | 剩余 120-170 款 | 页面上线即有 SEO 价值，边跑边加 |

## 七、待决策的开放问题

1. **子域名命名**：用 `battery.sgcarpass.com` 还是另取新域名？  
   - 子域名：快速上线，复用主域权重
   - 新域名：品牌独立，但 SEO 从零开始

2. **车型数据来源**：手动整理 vs 批量采集？  
   - 手动：质量高，初期 30 款可控
   - 批量：后期扩量需要，可以考虑爬 sgcarmart / 参考 LTA 数据

3. **修理工核销入口**：在现有后台做还是另起简单页面？  
   - 现有后台：改动小，但修理工需登录系统
   - 独立页面：体验更好，改动稍多

---

## 八、优先级建议

```
1. 内容优先   ← 先建 30 个车型页，没内容 SEO 无从谈
2. 领券流程   ← 复用现有 WA 基础设施，改动最小
3. 核销页面   ← 可先用现有后台手动操作，再优化体验
```

## 九、三个决策问题的最终解答（2026-05-15 补充）

### 9.1 WhatsApp 按钮用哪个号？

**结论：复用现有 81525879，不新开号。**

新开号需要重新过 Meta 审核、重新接 Webhook，工作量大没必要。

网页上的 WA 按钮用 **wa.me 预填文本链接**，无需改后端：

```
https://wa.me/6581525879?text=我想领取电瓶优惠券
```

车主点击跳转到 WA，发出预填文字。Dispatch 识别到这句话，走优惠券流程。  
现有修理工流程（注册/发码/查询）完全不受影响。

---

### 9.2 那么多车型怎么管理？需要建新表吗？

**结论：一个页面模板 + 一个数据文件，不是每个车型一个独立文件。**

Nuxt 3 动态路由，只写**一个页面文件**：

```
pages/car/[slug].vue   ← 这一个文件处理所有车型
```

访问 `/car/toyota-vios` 时 slug = `toyota-vios`，页面去取对应数据渲染。  
150 个车型共用同一个模板，维护一处即可。

**数据管理分两阶段：**

**第一阶段（先跑起来）：JSON 文件，不需要建新表**

在 Nuxt 项目里放 `data/cars.json`：

```json
[
  {
    "slug": "toyota-vios",
    "make": "Toyota",
    "model": "Vios",
    "yearFrom": 2014,
    "yearTo": 2023,
    "batteries": [
      { "partNo": "55D23L", "voltage": 12, "capacity": 45, "type": "EFB", "priceMin": 180, "priceMax": 220 }
    ],
    "faq": [
      { "q": "Toyota Vios battery how long?", "a": "2-3 years in Singapore's climate." }
    ]
  }
]
```

好处：不动后端，前端自己管内容，改数据不需要重启服务，git 直接追踪变更历史。

**第二阶段（内容多了再迁）：后端建 `car_model` 表**

等车型超过 50 条、需要后台可视化编辑时，再建表、写管理页面。  
届时 Nuxt 只改数据源，页面模板不用动。

---

### 9.3 优惠券和核销什么时候做？怎么做？

**结论：现在先不做，等 SEO 跑出流量再加。**

做优惠券系统的前提是有人看这个页面。页面没流量时先把核销系统做好，是倒着来的。

**分三阶段推进：**

| 阶段 | 触发条件 | 做什么 |
|---|---|---|
| 阶段一（现在） | — | 建 SEO 车型页，CTA 先用 WA 咨询链接替代 |
| 阶段二 | 每天 10+ 人通过 WA 按钮发消息 | 车主输手机号 → 后端生成券码 → WA 发给车主 |
| 阶段三 | 门店开始推广 | 修理工在现有后台输券码核销，记录佣金 |

**阶段二技术方案（到时候再做）：**
- 后端加一张 `tys_bw_promo_coupon` 表
- Dispatch 识别「我想领取电瓶优惠券」→ 进入领券流程
- WA 回复券码，复用现有 WA 基础设施

**阶段三核销方案（最简单的方式）：**
- 修理工在 WA 发「核销 BAT-7X3K」
- 完全复用现有 DispatchService，加一个新 case
- 不需要另起核销页面
